# Skill #8 — UI (Front Design)

> *"Los screen flows dicen qué. El front design dice cómo se siente."*

## Qué Hace Este Skill

Toma los screen flows del Skill #7 y los convierte en interfaces reales: componentes,
código, paletas, tipografía y composición visual. Aplica los principios del
`front-end-design.md` y usa el skill instalado `frontend-design:frontend-design` para
producir UI production-grade con identidad visual DISTINTIVA — no el template genérico
de Tailwind por defecto.

**Por qué va después de UI Design Workflow:**
El Skill #7 estableció QUÉ pantallas existen, QUÉ datos muestran y QUÉ acciones tienen.
El Skill #6 definió cómo se organiza el producto y cómo se comportan los controles.
Este skill toma todo eso como spec y decide CÓMO se ve: paleta, tipografía, atmósfera,
composición, motion. El resultado es UI implementada que cumple los acceptance targets
del Skill #7 y pasa el test anti-AI-slop.

## Workflow

### Paso 1: Design Thinking

**Leer:** `.claude/skills/la-herreria/assets/front-end-design.md` — aplicar sus principios.

Antes de escribir una línea de código, definir la dirección estética del producto.
Esta decisión aplica a TODAS las pantallas — es el DNA visual del producto.

```
DESIGN THINKING — [Nombre del Proyecto]

PROPÓSITO:
¿Qué problema resuelve? ¿Quién lo usa? ¿En qué contexto? ¿Desde qué dispositivo?

TONO ESTÉTICO — elegir UNA dirección BOLD:
→ Brutalmente minimal | Luxury/refined | Retro-futuristic
→ Organic/natural | Editorial/magazine | Playful/toy-like
→ Brutalist/raw | Art deco/geometric | Industrial/utilitarian
→ Soft/pastel | Neo-corporate | [Dirección propia]

DIFERENCIADOR VISUAL:
¿Qué hace este UI MEMORABLE? ¿Qué verá el usuario y no olvidará?
(Un detalle, una tipografía, un color, un patrón)

ANTI-PATTERNS a evitar en este proyecto:
✗ Inter/Roboto/Arial como fuente principal
✗ Gradiente morado sobre blanco
✗ Blue #3B82F6 Tailwind default como primario
✗ Cards sin personalidad propia
✗ Paleta "safe" de grises + 1 accent color tímido
```

**Discutir con el usuario:**
```
🎨 Vamos a definir la identidad visual de [nombre].

Antes de diseñar, 3 preguntas:

1. ¿Tienes colores de marca ya definidos?
   (Si no, propongo basándome en tu industria y audiencia)

2. ¿Qué SENSACIÓN quieres provocar?
   Piensa en: ¿Elegante como hotel boutique? ¿Eficiente como cockpit?
   ¿Cálida como cafetería artesanal? ¿Bold como marca streetwear?

3. ¿Alguna app o sitio cuyo diseño admires?
   (No tiene que ser del mismo rubro)
```

Si el usuario no tiene preferencias → proponer UNA dirección BOLD basada en
industria y audiencia. Nunca la opción "safe" por defecto.

---

### Paso 2: Definir el Design System + Generar DESIGN.md

Con la dirección estética clara, establecer el sistema de diseño que aplica
a todo el producto.

**Generar `DESIGN.md` en el root del proyecto** siguiendo el template de
`.claude/skills/impeccable/references/design-md-template.md` (8 secciones,
formato portable consumible por Claude Design y otros agentes). Este archivo
será el source of truth que consumen `/critique`, `/polish`, `/normalize` y
el Design Critic agent.

Además, documentar en detalle en `UI-[nombre].md` (ver abajo):

```markdown
## Design System — [Nombre del Proyecto]

### Paleta de Color
- Dominante: [color] [hex] — [qué evoca]
- Acento: [color] [hex] — [contraste intencional]
- Semánticos: Success [hex], Error [hex], Warning [hex], Info [hex]
- Neutrales: [escala de más claro a más oscuro]
- Fondo: [color + tratamiento — ¿textura? ¿gradiente? ¿tono cálido/frío?]
- Regla de dominancia: el dominante manda, el acento sorprende

### Tipografía
- Display/Headings: [Fuente DISTINTIVA — ej: Clash Display, Playfair Display, DM Serif Display]
  → Peso, tamaño, letter-spacing
  → Por qué esta fuente: [conexión con el tono]
- Body: [Fuente complementaria — ej: DM Sans, Outfit, Source Serif 4]
  → 16px, regular, line-height 1.5-1.6
- Mono (si aplica): [JetBrains Mono, Fira Code]
- PROHIBIDO: Inter, Roboto, Arial, system-ui como fuente principal

### Componentes Core
- Botones primarios: [color, radius, padding, hover effect con carácter]
- Botones secundarios: [outline/ghost — coherentes con el primario]
- Cards: [¿borde de color? ¿gradiente? ¿sombra dramática? ¿flat?]
- Inputs: [¿underline? ¿outlined? ¿filled? Focus state con personalidad]
- Iconos: [set específico — Lucide/Phosphor/Heroicons, estilo line/filled]
- Navigation: [implementar el patrón del Skill #6 — sidebar/top nav/bottom nav]
- Error states: [color, icono, tipografía — consistente con interaction-patterns.md]

### Layout & Composición
- Patrón de navegación: [del Skill #6 — top nav / sidebar / bottom nav]
- Desktop: [estructura + ¿algún elemento que rompe el grid?]
- Mobile: [derivado del patrón IA del Skill #6]
- Spacing: [¿generoso o denso?]

### Detalles de Atmósfera
- Fondo: [¿sólido? ¿gradiente mesh? ¿noise texture? ¿patrón?]
- Sombras: [¿dramáticas? ¿coloreadas? ¿sutiles? ¿flat?]
- Bordes: [¿delgados neutros? ¿gruesos coloridos? ¿sin bordes?]
- Motion: [¿staggered reveals? ¿hover states? ¿scroll-triggered?]

### Reglas de Comportamiento (del Skill #6)
- Feedback: [timings definidos en interaction-patterns.md]
- Validación de forms: [on blur, errores inline, focus en primer error]
- Empty states: [diseñados según onboarding.md — apuntan al first success]
```

### Paso 3: Implementar Screen por Screen

Para cada screen flow del Skill #7, implementar el código UI usando el skill
instalado `frontend-design:frontend-design`.

**Por cada pantalla:**

1. Leer el screen flow (layout, data, actions, states, acceptance targets)
2. Revisar los interaction patterns del Skill #6 relevantes para esa pantalla
3. Aplicar el design system establecido
4. **Invocar `frontend-design:frontend-design`** con el contexto completo:
   - Design system (Paso 2)
   - Screen flow de la pantalla
   - Acceptance targets
   - Interaction patterns aplicables
   - Patrón de navegación IA
5. Cubrir TODOS los estados: Default, Loading, Empty (según onboarding.md), Error, Success
6. Verificar los acceptance targets definidos en el Skill #7

**Orden de implementación recomendado:**
```
1. Layout shell + navegación (implementar el patrón del Skill #6 — establece look & feel global)
2. Pantalla principal / Dashboard (la más compleja, define el sistema)
3. Formulario principal (inputs, validaciones según interaction-patterns.md, botones)
4. Empty states (según onboarding.md — first success flow)
5. Vistas de listado / tabla
6. Vistas de detalle con state transitions (según interaction-patterns.md)
7. Pantallas de auth (login/signup — simples pero con el mismo design system)
8. Pantallas secundarias (settings, perfil, etc.)
```

**Razón:** Empezar por el layout y la pantalla más compleja porque establece
el sistema visual. Todo lo que sigue hereda ese contexto.

---

### Paso 4: Componentes Nuevos

Para los componentes nuevos identificados en el Skill #7
(`docs/ui-design/components/[nombre].md`), implementar el componente en
`src/shared/ui/[ComponentName].tsx` siguiendo la spec definida.

**Invocar `frontend-design:frontend-design`** para cada componente nuevo,
proveyendo la spec del componente y el design system establecido.

**Reglas de implementación de componentes:**
- Seguir la spec (props, states, accessibility, variants)
- Tipado TypeScript completo
- Variantes con CVA (class-variance-authority) si hay múltiples variantes
- Accesibilidad: role, aria-label, keyboard navigation, contraste WCAG AA
- Estados: default, hover, focus, disabled, error, loading

---

### Paso 5: Anti-AI-Slop Check

Antes de considerar el skill completado, aplicar el test mental:

```
✅ ¿Si pongo 10 apps generadas por IA en una fila, esta se distingue? → SÍ
✅ ¿La tipografía tiene personalidad propia? → No es Inter/Roboto/Arial
✅ ¿La paleta tiene DOMINANCIA? → Un color manda, no colores equidistantes
✅ ¿Las cards, botones y inputs tienen identidad PROPIA? → No son shadcn defaults
✅ ¿Hay al menos UN detalle que alguien va a recordar? → [identificarlo]
✅ ¿Los estados empty y error fueron diseñados? → No son placeholders genéricos
✅ ¿El motion tiene intención? → No son bounces arbitrarios
✅ ¿La navegación refleja el patrón del Skill #6? → Coherencia con la IA definida
✅ ¿Los interaction patterns del Skill #6 están implementados? → Comportamiento consistente
✅ ¿Los empty states guían al first success? → Según onboarding.md del Skill #6
```

Si algún check falla → revisar el design system y corregir antes de entregar.

## Reglas Críticas

- **Leer front-end-design.md ANTES de cualquier decisión visual.** Sin excepción.
- **Usar `frontend-design:frontend-design` para implementar cada pantalla y componente.**
  Este skill produce UI production-grade con identidad visual distintiva.
- **Nunca usar Inter, Roboto o Arial como fuente principal.** Son la señal #1 de AI slop.
- **El design system se define UNA VEZ y aplica a TODAS las pantallas.** Sin inconsistencias.
- **Implementar TODOS los estados.** Empty state (según onboarding.md del Skill #6), error
  state (según interaction-patterns.md del Skill #6), loading state. No solo el happy path.
- **Los acceptance targets del Skill #7 son requisitos, no sugerencias.** Si un target
  falla, la pantalla no está completa.
- **El dispositivo primario lo definen las personas del Skill #4.** Si la persona es
  mobile-first, la implementación es mobile-first.
- **El patrón de navegación lo definió el Skill #6.** No cambiarlo ni mezclarlo.
- **Los interaction patterns del Skill #6 son producto-wide.** Todos los formularios
  validan igual. Todos los errores se presentan igual. Todos los estados de carga se ven igual.
